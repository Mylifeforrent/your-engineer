# 一、哪些地方是合理的

## 1. 不是单纯做聊天机器人，而是嵌入实际工作流

这是整个想法最合理的地方。

你没有把 AI 定位成“问答助手”，而是让它参与实际交付流程：

- 分析需求
- 拆分任务和 User Story
- 创建 Jira
- 更新 Confluence
- 执行测试
- 汇总发布材料
- 分析生产事故

这种产品比单纯的 AI 对话工具更容易产生可量化价值，因为结果直接进入现有工作系统。

评价产品价值时，可以直接观察：

- 需求分析耗时降低多少
- Jira 创建时间减少多少
- Release 文档准备时间减少多少
- 事故初步定位时间缩短多少
- 用户接受 AI 建议的比例是多少

这是正确的产品方向。

---

## 2. 使用“业务领域知识”增强需求分析是合理的

用户选择业务领域，然后系统加载相应的业务知识，比让 AI 直接分析一段孤立需求可靠得多。

例如同样一句“增加退款功能”，不同领域的分析完全不同：

- 电商领域需要考虑订单状态、库存恢复、支付渠道
- 金融领域需要考虑审批、风控、会计记账和合规
- 订阅业务需要考虑订阅周期、按比例退款和权益回收

因此，领域上下文是需求分析模块的核心资产。

但这里应进一步把“业务领域”设计成更完整的 **Workspace / Project Context**，不仅包含业务知识，还应包括：

- 术语表
- 系统架构
- 业务流程
- 数据模型
- 历史需求
- Jira 项目
- Confluence 空间
- 团队规范
- Definition of Ready
- Definition of Done
- User Story 模板
- 验收标准模板

这样 AI 才是在团队环境中工作，而不是简单地挂载几篇知识文档。

---

## 3. AI 生成后由用户确认再执行是正确设计

你提到：

- AI 给出建议
- 用户可以修改
- 用户确认后创建 Jira
- 用户确认后同步 Confluence

这是非常重要的 Human-in-the-loop 机制。

尤其是 Jira、Confluence、生产系统相关操作，第一阶段不应该允许 AI 静默执行。合理流程应是：

1. AI 生成草稿
2. 用户查看来源和推理依据
3. 用户修改
4. 系统进行格式和权限校验
5. 用户确认
6. 通过确定性的 API 执行操作
7. 保存审计记录

AI 应负责理解、组织和生成内容，真正的系统操作应由确定性的连接器完成。

---

## 4. 将测试结果、日志和截图作为可审查证据是合理的

测试模块中保留以下内容是必要的：

- 执行状态
- 开始和结束时间
- 执行日志
- 错误堆栈
- 截图
- 视频或 Trace
- 测试环境
- 执行版本
- 参数
- 触发方式

你强调“无论报错还是失败，用户都能看到实际执行情况”，这是正确的。测试平台不能只输出一个 AI 总结，因为用户必须能够看到底层证据。

AI 可以在这些证据之上提供：

- 失败摘要
- 可能原因
- 相似历史失败
- 建议排查步骤
- 是否可能是 flaky test
- 是否可能由环境问题导致

但不能替代原始日志。

---

## 5. Release 准备工具是一个价值较高的切入点

Release 文档通常需要人工从多个系统中搜集信息：

- Jira 完成项
- 未完成项
- PR 列表
- 测试结果
- 已知问题
- 风险项
- 回滚方案
- 变更说明
- 部署步骤

这类工作具有几个特点：

- 重复性高
- 数据来源相对结构化
- 输出格式稳定
- 风险低于自动执行生产操作
- 效果容易量化

因此，Release 准备工具很适合作为早期 MVP 功能。

合理流程可以是：

> 选择 Release 版本 → 读取 Jira、PR、测试结果、Confluence → 检查缺失项 → 生成 Release 草稿 → 用户确认 → 发布到 Confluence

这比一开始就做完整的生产告警分析更容易交付价值。

---

## 6. Alert Analyzer 的业务价值确实很高

生产问题排查通常需要人工完成：

- 判断影响哪个服务
- 查找相关日志
- 查看指标
- 查看 Trace
- 对照最近变更
- 查找类似历史事故
- 阅读 Runbook
- 给出初步结论

这些工作非常适合使用 AI 做信息聚合和初步分析。

你的方向——将系统、服务、日志来源、影响范围和历史修复经验配置起来——是正确的。

但这个模块应该被视为后期的高价值、高风险能力，而不是普通的小工具。

# 二、哪些地方不合理或需要优化

# 1. 当前范围不适合“轻量级服务”

从产品边界看，你现在描述的系统实际上包括：

| 能力 | 本质 |
|---|---|
| 需求分析 | AI 产品需求助手 |
| Jira/Confluence 操作 | 企业集成与工作流平台 |
| 测试脚本执行 | 自动化测试执行平台 |
| 定时健康检查 | 调度与监控平台 |
| Release 准备 | 发布管理平台 |
| Alert Analyzer | AIOps / Incident Intelligence 平台 |

每一个方向都有独立的权限模型、数据模型、运行模型和运维成本。

所以不合理的不是这些需求本身，而是把它们都放进第一阶段。

建议先确立一个单一的核心价值主张，例如：

> 帮助研发团队把需求、测试证据和交付信息快速转化为规范的 Jira、Confluence 和 Release 文档。

这比“覆盖研发全过程的 AI 平台”更容易落地。

---

# 2. 用户角色过多，产品主用户不清晰

当前系统同时服务于：

- 产品经理
- 业务分析师
- 开发人员
- QA
- Release Manager
- SRE
- Engineering Manager

这些角色的问题和使用频率不同。

例如：

- 产品经理关注需求完整性和 Jira 拆分
- QA 关注测试执行和失败定位
- SRE 关注日志、指标和事故响应
- Release Manager 关注发布证据和审批

如果没有明确主用户，界面和工作流会迅速变得复杂。

第一阶段必须明确：

- 谁是每天使用系统的人
- 他最频繁的一个任务是什么
- 当前完成这个任务需要多少时间
- 系统能减少哪几个步骤

从你目前的描述看，比较合理的首批用户是：

> QA Lead、研发项目负责人或 Release Owner。

因为他们通常同时接触 Jira、Confluence、测试结果和发布流程，能够跨多个模块验证产品价值。

---

# 3. “上传任意测试脚本并运行”风险非常高

这是当前方案中技术和安全风险最大的一点。

用户上传 Playwright 或 Pytest 脚本，本质上是在你的服务器上运行任意代码。它可能：

- 读取服务器文件
- 访问内网系统
- 获取环境变量
- 窃取 Token
- 发起无限循环
- 消耗大量 CPU 或内存
- 安装恶意依赖
- 调用未经授权的外部接口
- 对测试环境执行破坏性操作

即使只是团队内部使用，也不能忽略这些问题。

### 更合理的第一阶段方案

不要自行建设测试执行平台，而是优先连接现有执行基础设施，例如：

- Jenkins
- GitHub Actions
- GitLab CI
- Azure DevOps Pipeline
- 已有的 Kubernetes Job
- 团队现有测试服务器

你的系统负责：

1. 选择测试任务
2. 调用已有 CI/CD Pipeline
3. 获取执行状态
4. 拉取日志、截图和测试报告
5. 用 AI 分析测试失败
6. 将结果同步到 Jira 或 Confluence

这样可以显著降低安全风险和建设成本。

### 后续确实要自行执行脚本时

至少需要：

- 隔离容器或临时虚拟机
- 只读基础镜像
- CPU、内存和时间限制
- 网络访问白名单
- Secret Vault
- 文件系统隔离
- 依赖锁定
- 镜像扫描
- 完整审计日志
- 任务终止机制
- 并发和配额控制

这已经不是轻量级功能了。

---

# 4. “日常健康状态检查”不应该成为独立模块

它和测试用例模块的底层能力高度重复：

- 执行脚本
- 设置环境
- 调度任务
- 保存执行结果
- 产生截图和日志
- 发送通知

两者的区别主要是触发方式：

- 测试用例模块：用户手动触发
- 健康检查：定时触发
- 后续还可能有 API、Webhook、代码合并等触发方式

因此更合理的产品模型是统一建设一个 **Automation Run / Execution 模块**。

一个自动化任务可以配置：

- 执行内容
- 执行环境
- 触发方式
- 参数
- 超时时间
- 重试策略
- 结果处理方式
- 通知策略
- Jira/Confluence 同步规则

触发方式可以是：

- 手动
- 定时
- API
- Webhook
- PR 合并后
- Release 前
- 监控告警后

这样底层逻辑统一，避免维护两套相似系统。

---

# 5. 通知机制需要避免噪音

健康检查每次成功或失败都通知用户，最终很可能造成通知疲劳。

应支持通知策略，例如：

- 只在失败时通知
- 只在状态变化时通知
- 连续失败两次才通知
- 达到某个失败比例才通知
- 恢复正常时通知
- 同类问题在一定时间内合并
- 夜间只通知高优先级问题

还应支持：

- 严重级别
- 通知分组
- 静默时间
- 升级策略
- 负责人
- 去重规则

否则健康检查模块很容易变成新的噪音来源。

---

# 6. “小工具模块”容易发展成无边界功能集合

Jira Review、Jira 创建、Confluence 修改、Release 准备等功能都可以称作小工具，但这种分类方式不利于长期产品设计。

随着用户提出更多需求，系统可能出现几十个按钮：

- 创建 Bug
- 创建 Story
- 更新 Epic
- 总结 Sprint
- 修改 Confluence
- 生成 Release Note
- 生成测试报告
- Review Jira
- Review PR
- 创建 Incident

最后产品会变成一个无结构的工具箱。

更合理的方式是建设一个 **模板化工作流引擎**，由以下部分组成：

- 输入源
- AI 指令
- 输出结构
- 审批步骤
- 执行动作
- 目标系统
- 权限规则

例如“生成 Release 文档”实际上是一个工作流：

```text
输入：
Jira Release + PR + 测试结果

处理：
汇总变更、识别风险、检查缺失项

输出：
Release Document

动作：
用户确认后写入 Confluence
```

“Jira Review”也是一个工作流：

```text
输入：
Jira Issue

处理：
检查描述、验收标准、依赖、风险

输出：
Review 建议

动作：
用户选择是否更新 Jira
```

产品层面应展示为场景化工具，底层则使用统一工作流能力。

---

# 7. 不应让 AI 直接承担确定性 CRUD 操作

例如“创建 Jira”本身不需要 AI。

AI 适合做的是：

- 理解用户意图
- 生成描述
- 拆分任务
- 补全验收标准
- 判断字段应该如何填写
- 提示缺失信息

真正执行 Jira 创建时，应使用确定性逻辑：

- 校验 Project Key
- 校验 Issue Type
- 校验必填字段
- 校验用户权限
- 映射自定义字段
- 调用 Jira API
- 检查返回值
- 保存审计记录

原则应是：

> AI 负责非结构化理解，程序负责结构化执行。

否则一旦 AI 生成错误字段、错误项目或错误用户，故障会很难追踪。

---

# 8. 第三方配置中心的复杂度被低估了

“配置 Jira 和 Confluence”表面上只是保存 URL 和 Token，但实际需要考虑：

- OAuth 还是 API Token
- 用户级凭证还是团队级凭证
- 凭证由谁创建
- 谁可以使用
- 谁可以修改
- 凭证如何加密
- 凭证过期如何处理
- 是否支持多个 Jira 实例
- 项目和空间如何映射
- 自定义字段如何映射
- 如何测试连接
- 如何记录操作审计
- 离职用户的凭证如何迁移

配置中心至少需要包含：

### Integration

具体连接，例如 Jira、Confluence、GitHub、日志平台。

### Credential

经过加密管理的 Token 或 OAuth 授权。

### Mapping

业务领域与 Jira Project、Confluence Space、Repository 的对应关系。

### Policy

允许执行哪些操作，是否必须人工审批。

### Audit

谁在什么时间对哪个系统执行了什么操作。

内部使用可以减少商业化要求，但不能降低安全要求。

---

# 9. 需求分析不能只输出 Task 和 User Story

需求分析的价值不应只是把一段文字改写成 Jira 格式。

更有价值的分析结果应包含：

- 需求目标
- 用户角色
- 用户场景
- 业务价值
- 范围内事项
- 范围外事项
- 功能需求
- 非功能需求
- 验收标准
- 业务规则
- 异常场景
- 数据要求
- 权限要求
- 依赖系统
- 风险
- 未决问题
- 建议的 Epic、Story 和 Task
- 建议测试范围

AI 应先识别信息缺口，而不是立即生成大量看起来完整的 Jira。

例如，需求缺少以下信息时，应明确提示：

- 谁可以执行
- 是否允许重复操作
- 失败后如何恢复
- 是否需要审计
- 数据保存多久
- 是否存在性能要求
- 是否需要兼容旧流程

“提出正确的问题”往往比“直接生成 User Story”更有产品价值。

---

# 10. 超链接输入需要更严格的边界

让 AI 访问用户输入的链接是合理的，但需要处理：

- 链接是否需要登录
- 用户是否有访问权限
- 页面是否允许抓取
- 页面是否使用动态渲染
- 页面内容是否过长
- 页面内容是否更新
- 是否是 Jira、Confluence、Google Docs 或普通网页
- 页面中是否包含敏感信息
- 是否可能触发 SSRF 等安全问题

产品上不应简单写成“AI 会访问链接”。

更合理的是：

1. 系统识别链接类型
2. 使用对应连接器获取内容
3. 展示读取成功或失败状态
4. 保存内容来源和更新时间
5. 在 AI 输出中标记引用来源
6. 对无法访问的内容明确提示

AI 分析结果最好能够指出：

> 该建议来自需求描述、某个 Confluence 页面或某条历史需求。

否则用户无法判断结论是否可信。

---

# 11. Alert Analyzer 第一阶段不能自动下结论或自动修复

生产事故分析的风险远高于需求分析。

AI 很可能面对：

- 日志不完整
- 指标延迟
- 服务依赖关系过期
- 同时发生多个问题
- 历史事故与当前事故表面相似但根因不同

因此第一阶段应定位为：

> Read-only Incident Triage Assistant

输出内容应包括：

- 当前现象
- 影响范围
- 已检查的数据源
- 关键证据
- 可能原因列表
- 每个原因的置信度
- 尚未确认的信息
- 推荐的下一步检查
- 相关历史事故
- 最近的部署或配置变更

不应在早期阶段：

- 自动重启服务
- 自动回滚
- 自动修改配置
- 自动关闭告警
- 将猜测描述为确定根因

即使未来支持自动修复，也应建立审批、回滚和安全策略。

# 最终判断

你的核心思路是合理的，尤其合理的部分包括：

- 使用业务上下文增强 AI
- 嵌入 Jira 和 Confluence 工作流
- 保留人工确认
- 将日志、截图等原始证据展示给用户
- 用 AI 减少 Release 和事故分析中的信息聚合工作

需要重点优化的部分是：

1. **缩小首期范围。**
2. **明确首批核心用户。**
3. **测试执行与健康检查共用统一执行引擎。**
4. **第一阶段不要自行运行任意用户脚本。**
5. **把“小工具”升级为模板化工作流，而不是不断增加孤立功能。**
6. **AI 负责理解，确定性系统负责执行。**
7. **Alert Analyzer 先做只读辅助分析。**
8. **从一开始建设权限、凭证、审计和结果追溯能力。**

较稳妥的产品路线是：

> **第一阶段：需求分析 + Jira/Confluence + Release 准备**  
> **第二阶段：接入现有测试执行系统 + 健康检查**  
> **第三阶段：Alert Analyzer 和更高级的生产运维能力**

这条路径能保留你的整体愿景，同时避免第一阶段直接陷入平台级复杂度。

用户:
我目前最希望先解决的几个点：
1. release 准备
2. 告警分析

这两块优先实现，因为目前最花费时间。

需求分析放在后面。自动化测试执行优先级比需求分析高一点点。

帮我按照这个调整优先级
AI助手:


AI助手:
可以。按照你目前最迫切的痛点，产品优先级应调整为：

> **Release 准备 → 告警分析 → 自动化测试执行 → 需求分析**

但需要区分两类优先级：

- **业务功能优先级**：用户首先看到和使用哪些功能。
- **基础能力建设顺序**：为了支撑这些功能，底层必须先完成哪些能力。

否则容易出现“先开发告警分析界面，但日志、权限、服务配置和审计能力都不存在”的问题。

# 一、调整后的整体优先级

| 优先级 | 模块 | 业务价值 | 实现复杂度 | 风险 | 建议定位 |
|---|---|---:|---:|---:|---|
| P0 | Release 准备 | 很高 | 中 | 中 | 首个 MVP |
| P1 | Alert Analyzer | 很高 | 很高 | 很高 | 第二个核心模块 |
| P2 | 自动化测试执行与分析 | 高 | 高 | 高 | 第三阶段 |
| P3 | 需求分析 | 中高 | 中 | 中 | 后续阶段 |
| 基础能力 | 配置、连接器、权限、审计 | 必需 | 中高 | 高 | 贯穿所有阶段 |

“配置中心”不再作为一个独立业务模块排优先级，而是作为 Release 和告警分析必须依赖的平台能力逐步建设。

# 二、第一优先级：Release 准备

## 1. 产品目标

核心目标不是单纯“生成一篇 Release 文档”，而是：

> 自动收集发布相关信息，识别发布准备中的缺失项和风险，并生成可审查、可追溯的 Release 文档。

它需要解决三个主要问题：

1. 信息分散在 Jira、Confluence、PR、测试报告等多个系统。
2. 人工整理重复且耗时。
3. 发布前容易遗漏风险、测试证据、回滚方案等信息。

---

## 2. 推荐的 MVP 流程

```text
选择 Release
    ↓
读取 Jira Issues
    ↓
读取 PR / Commit 信息
    ↓
读取已有测试结果
    ↓
读取 Release 模板
    ↓
AI 汇总并检查缺失项
    ↓
生成 Release 文档草稿
    ↓
用户确认和修改
    ↓
发布到 Confluence
```

第一版不需要做高度自由的流程编排，先固定一个标准工作流。

---

## 3. 第一版输入范围

建议支持以下信息源：

### 必须支持

- Jira Release、Fix Version 或指定 Issue 列表
- Confluence Release 模板
- 用户补充说明
- 手动上传测试报告或测试结果摘要

### 后续支持

- GitHub、GitLab、Bitbucket PR
- Jenkins、GitHub Actions、GitLab CI
- 自动化测试执行结果
- 代码扫描和安全扫描结果
- Incident 和 Known Issue

第一版不需要一次性连接所有平台，可以根据团队当前实际工具选择最重要的两到三个系统。

---

## 4. AI 应完成的工作

AI 可以负责：

- 汇总本次发布内容
- 按业务功能分组
- 生成 Release Summary
- 生成变更列表
- 识别高风险变更
- 总结测试执行情况
- 识别缺失测试证据
- 识别未关闭 Jira
- 识别未合并或未审批的 PR
- 生成已知问题
- 生成回滚注意事项草稿
- 生成发布后检查项
- 按模板生成 Confluence 内容

AI 不应该直接决定：

- 是否允许发布
- 某个风险是否可以接受
- 测试是否足够
- 是否执行部署
- 是否执行回滚

这些应该由用户或确定性规则决定。

---

## 5. Release 准备模块的核心输出

建议不要只输出文档，还要输出一个结构化的 Release Readiness Report。

### Release 概览

- Release 名称
- 版本
- 环境
- 计划发布时间
- 发布负责人
- 涉及服务

### 变更内容

- 新功能
- Bug 修复
- 配置变更
- 数据库变更
- 依赖变更

### 测试证据

- 测试范围
- 自动化测试结果
- 手工测试结果
- 未测试内容
- 已知失败

### 风险

- 高风险 Jira
- 未关闭 Issue
- 未审批 PR
- 缺少测试证据
- 缺少回滚方案
- 影响关键服务的变更

### 发布检查

- 发布前检查
- 发布步骤
- 发布后验证
- 回滚条件
- 回滚步骤

### AI 分析状态

每条内容标记为：

- 已从来源确认
- 根据多项信息推断
- 用户手动输入
- 当前缺失
- 需要人工确认

这会比单纯生成一篇自然语言文档更可靠。

---

## 6. Release 模块的 MVP 范围

第一版建议只实现：

1. 选择 Jira Version 或输入 Jira Issue。
2. 读取 Jira 内容。
3. 允许用户上传或填写测试结果。
4. 使用团队 Release 模板生成草稿。
5. AI 检查缺失项。
6. 用户修改和确认。
7. 创建或更新 Confluence 页面。
8. 保存操作记录和生成来源。

第一版可以暂时不做：

- 自动部署
- 自动发布审批
- 自动触发回滚
- 自定义工作流设计器
- 复杂的多阶段 Release
- 多团队依赖编排

---

## 7. Release 模块成功指标

建议重点统计：

- 准备 Release 文档的平均耗时
- 自动填充内容比例
- 用户对 AI 内容的修改比例
- 发布前发现的缺失项数量
- Release 文档被实际采用的比例
- Jira、测试结果和文档之间的一致性
- Release 准备过程中减少的人工步骤数量

# 三、第二优先级：Alert Analyzer

Alert Analyzer 的业务价值很高，但风险和技术复杂度明显高于 Release 准备，因此建议在 Release MVP 之后重点投入。

## 1. 第一阶段产品定位

第一阶段建议明确定位为：

> **只读的生产告警初步分析助手。**

它负责搜集证据、关联信息和给出排查建议，但不直接修改生产系统。

第一版应避免：

- 自动重启服务
- 自动扩容
- 自动修改配置
- 自动回滚
- 自动关闭告警
- 自动判定根因并向外正式发布

---

## 2. 告警分析的推荐流程

```text
接收告警
    ↓
识别系统、服务和环境
    ↓
读取服务配置与依赖关系
    ↓
读取相关日志、指标和变更记录
    ↓
检索历史相似事件和 Runbook
    ↓
AI 进行证据汇总和假设排序
    ↓
生成初步分析报告
    ↓
通过 Email 或 API 通知用户
```

---

## 3. 第一版输入来源

不要第一版就接入所有可观测性平台。

建议先选择团队使用频率最高的数据源。

### 必须支持

- 告警内容
- 服务基本信息
- 服务负责人
- 日志访问方式
- 最近部署记录
- 历史故障记录
- Runbook 或排障文档

### 后续支持

- Metrics
- Distributed Trace
- Kubernetes Events
- Cloud Events
- Database Slow Query
- Feature Flag 变化
- 配置中心变更
- 网络和基础设施事件

---

## 4. 告警输入需要结构化

告警不能只是一段自然语言。建议至少建立以下字段：

- Alert ID
- Alert Type
- Severity
- Environment
- System
- Service
- Region
- Start Time
- Current Status
- Triggered Metric
- Threshold
- Alert Message
- Dashboard Link
- Log Query
- Related Release

系统收到告警后，应先做结构化解析，再交给 AI。

---

## 5. AI 输出结构

告警分析结果建议固定包含：

### 事件概览

- 当前发生了什么
- 开始时间
- 受影响系统
- 可能影响范围
- 当前严重程度

### 关键证据

- 相关错误日志
- 指标异常
- 最近部署
- 配置变更
- 上游或下游异常
- 相似历史事件

### 原因假设

每个假设包含：

- 可能原因
- 支持证据
- 反向证据
- 置信度
- 仍需检查的信息

例如：

| 假设 | 支持证据 | 置信度 |
|---|---|---:|
| 最近发布导致接口错误 | 发布后错误率立即上升 | 高 |
| 下游服务超时 | 日志存在大量 timeout | 中 |
| 数据库连接池耗尽 | 部分日志相关，但缺少指标证据 | 低 |

### 推荐下一步

- 建议查询什么日志
- 建议查看什么指标
- 建议联系哪个团队
- 建议确认哪个变更
- 建议是否考虑回滚

### 信息缺口

系统必须明确说明当前缺少什么，而不是强行给出结论。

---

## 6. Alert Analyzer MVP 建议进一步收窄

第一版只支持：

- 一个或少数几个核心系统
- 一个日志平台
- 一个部署信息来源
- 一套固定的告警类型
- Email 或 API 输出
- 只读分析

例如第一版可以只解决：

> 某个核心服务出现接口错误率或延迟告警后，自动查询相关日志、最近发布和历史故障，输出初步诊断。

不要第一版支持所有系统和所有类型的告警。

---

## 7. 告警分析的关键风险控制

### 权限隔离

用户只能分析自己有权限访问的系统和日志。

### 敏感数据处理

日志中可能包含：

- 用户数据
- Token
- Cookie
- 邮箱
- 手机号
- 支付信息

在发送给模型前应进行脱敏和过滤。

### 证据引用

每个结论应该能够关联到原始日志、指标或变更记录。

### 不确定性表达

系统需要区分：

- 已确认事实
- 高概率推断
- 低概率猜测
- 尚未获取的信息

### 分析超时和降级

部分日志平台不可用时，应输出：

> 当前无法访问某数据源，因此分析结果不完整。

不能在证据缺失时表现得非常确定。

---

## 8. Alert Analyzer 成功指标

建议统计：

- 从收到告警到生成初步分析的时间
- Top 3 原因包含真实根因的比例
- 推荐排查步骤的有效率
- 用户查看分析后的下一步操作
- 重复告警识别率
- 相似历史 Incident 命中率
- 无依据结论比例
- 告警平均定位时间降低程度

其中最重要的不是“AI 是否准确指出唯一根因”，而是：

> AI 是否明显减少了人工搜集信息和初步排查的时间。

---

# 四、第三优先级：自动化测试执行与分析

自动化测试优先级高于需求分析，但建议拆成两个阶段。

## 第一阶段：接入已有测试执行系统

优先支持：

- 触发 Jenkins、GitHub Actions 或现有 CI Pipeline
- 查看执行状态
- 获取日志
- 获取截图和报告
- AI 总结失败原因
- 将测试结果提供给 Release 模块

这一步能够直接增强 Release 准备模块。

推荐形成以下关系：

```text
自动化测试执行
       ↓
产生测试证据
       ↓
Release 准备读取测试证据
       ↓
AI 判断发布材料是否完整
```

## 第二阶段：自建测试执行能力

只有在现有 CI 无法满足需求时，再考虑：

- 上传 Playwright 和 Pytest 脚本
- 容器化执行
- 环境和 Secret 管理
- 定时运行
- 资源配额
- 网络访问控制

因此，第三优先级的第一步不是“运行任意上传脚本”，而是“整合已有测试基础设施”。

---

# 五、第四优先级：需求分析

需求分析保留在产品规划中，但暂不作为当前重点。

后续实施时，可以复用已经建设的能力：

- Workspace
- 业务知识库
- Jira 连接器
- Confluence 连接器
- 模板管理
- AI 内容编辑
- 权限与审计

因此，需求分析放在后面并不会浪费前期建设。

实际上，Release 和 Alert Analyzer 会先帮助你建立更关键的企业内部能力：

- 多数据源读取
- 权限控制
- 内容溯源
- 结构化 AI 输出
- 人工确认
- 第三方系统写入
- 审计日志

这些能力以后都能复用于需求分析。

## 阶段 0：共享基础能力

需要实现：

- 用户和团队
- Workspace

- Credential 安全存储

- AI 模型调用层
- 模板管理

- 审计日志

- 来源引用

只实现当前 Release 功能需要的最小范围，不建设通用低代码平台。

---

## 阶段 1：Release Preparation MVP

核心范围：

- 从 Jira 读取 Release 信息
- 收集测试结果
- 按模板生成 Release 文档
- 检查缺失项和风险
- 用户确认
- 同步 Confluence
- 保存版本和审计记录

阶段目标：

> 让团队可以真正使用这个系统完成一次实际 Release 准备。

---

## 阶段 2：Alert Analyzer MVP

核心范围：

- 接收固定类型告警
- 识别服务
- 查询日志
- 关联最近发布
- 查询历史 Incident 和 Runbook
- 生成结构化分析
- Email 或 API 通知
- 保存分析证据

阶段目标：

> 在有限服务和有限告警类型下，明显缩短初步排查时间。

---

## 阶段 3：测试执行与分析

核心范围：

- 接入现有 CI/CD
- 手动或定时触发测试
- 获取日志和截图
- AI 分析失败
- 将结果同步到 Release 模块
- 根据通知策略发送结果

阶段目标：

> 打通测试证据与 Release 准备之间的数据链路。

---

## 阶段 4：需求分析

核心范围：

- 输入文字或链接
- 选择业务领域
- 检查需求缺口
- 生成 User Story 和验收标准
- 创建 Jira
- 同步 Confluence

# 八、两个核心模块之间的协同关系

Release 准备和 Alert Analyzer 并不是两个完全独立的模块。

它们可以共享很多数据：

| 共享数据 | Release 使用方式 | Alert Analyzer 使用方式 |
|---|---|---|
| Jira | 获取变更内容 | 查询关联 Issue |
| PR / Commit | 生成变更摘要 | 判断最近变更 |
| 部署记录 | 描述发布内容 | 判断故障是否与发布有关 |
| 测试结果 | 判断发布准备状态 | 判断问题是否曾被测试覆盖 |
| 服务目录 | 判断影响服务 | 查询服务负责人和依赖 |
| 历史 Incident | 提示已知风险 | 检索相似故障 |
| Confluence | 生成发布文档 | 获取 Runbook 和历史方案 |

因此，建议在数据模型上提前设计统一实体：

- Workspace
- Service
- Release
- Change
- Test Run
- Alert
- Incident
- Knowledge Document
- Integration
- Evidence

这样后续模块之间可以直接关联，而不是各自建立一套独立数据。

# 最终调整结果

新的优先级可以正式确定为：

## P0：Release Preparation

先解决信息收集、缺失检查、风险汇总和 Release 文档生成。

## P1：Alert Analyzer

以只读、有限服务、有限告警类型的方式实现初步诊断。

## P2：Automated Testing

先接入已有 CI/CD 和测试结果，再考虑自建脚本运行环境。

## P3：Requirement Analysis

复用 Jira、Confluence、知识库、模板和 AI 工作流能力，在后续阶段实施。

同时，配置中心、权限、连接器和审计不作为单独业务功能，而是随着 P0、P1 逐步建设。

这个顺序更符合你的实际痛点，也能形成比较清晰的产品演进路径：

> **发布前提高准备效率 → 生产中提高故障定位效率 → 用自动化测试增强发布证据 → 最后向需求阶段延伸。**
